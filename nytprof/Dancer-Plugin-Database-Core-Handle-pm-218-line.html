93Given a table name and a hashref of where clauses (see below for explanation),

95row as a hashref if called in scalar context, or a list of matching rows
96as hashrefs if called in list context. The third argument is a hashref of
97options to allow additional control, as documented below. For backwards
98compatibility, it can also be an arrayref of column names, which acts in the
99same way as the C<columns> option.
100
101The options you can provide are:
102
103=over 4
104
105=item C<columns>
106
107An arrayref of column names to return, if you only want certain columns returned
108
109=item C<order_by>
110
111Specify how the results should be ordered. This option can take various values:
112
113=over 4
114
115=item * a straight scalar or arrayref sorts by the given column(s):
116
117 { order_by => 'foo' } # equivalent to "ORDER BY foo"
118 { order_by => [ qw(foo bar) ] } # equiv to "ORDER BY foo,bar"
119
120=item * a hashref of C<order => column name>, e.g.:
121
122 { order_by => { desc => 'foo' } } # equiv to ORDER BY foo DESC
123 { order_by => [ { desc => 'foo' }, { asc => 'bar' } ] }
124 # above is equiv to ORDER BY foo ASC, bar DESC

126=back

129

131query. If called in scalar context, an implicit LIMIT 1 will be added to the
132query anyway, so you needn't add it yourself.

432# Given either a column name, or a hashref of e.g. { asc => 'colname' },
433# or an arrayref of either, construct an ORDER BY clause (quoting col names)
434# e.g.:
435# 'foo' => ORDER BY foo
436# { asc => 'foo' } => ORDER BY foo ASC
437# ['foo', 'bar'] => ORDER BY foo, bar
438# [ { asc => 'foo' }, { desc => 'bar' } ]
439# => 'ORDER BY foo ASC, bar DESC
440sub _build_order_by_clause {
441 my ($self, $in) = @_;
442
443 # Input could be a straight scalar, or a hashref, or an arrayref of either
444 # straight scalars or hashrefs. Turn a straight scalar into an arrayref to
445 # avoid repeating ourselves.
446 $in = [ $in ] unless ref $in eq 'ARRAY';
447
448 # Now, for each of the fields given, add them to the clause
449 my @sort_fields;
450 for my $field (@$in) {
451 if (!ref $field) {
452 push @sort_fields, $self->_quote_identifier($field);
453 } elsif (ref $field eq 'HASH') {
454 my ($order, $name) = %$field;
455 $order = uc $order;
456 if ($order ne 'ASC' && $order ne 'DESC') {
457 die "Invalid sort order $order used in order_by option!";
458 }
459 # $order has been checked to be 'ASC' or 'DESC' above, so safe to
460 # interpolate
461 push @sort_fields, $self->_quote_identifier($name) . " $order";
462 }
463 }
464
465 return "ORDER BY " . join ', ', @sort_fields;
466}

486=head1 WHERE clauses as hashrefs
487
488C<quick_update>, C<quick_delete> and C<quick_select> take a hashref of WHERE
489clauses. This is a hashref of field => 'value', each of which will be
490included in the WHERE clause used, for instance:
491
492 { id => 42 }
493
494Will result in an SQL query which would include:
495
496 WHERE id = 42
497
498When more than one field => value pair is given, they will be ANDed together:
499
500 { foo => 'Bar', bar => 'Baz' }
501
502Will result in:
503
504 WHERE foo = 'Bar' AND bar = 'Baz'
505
506(Actually, parameterised queries will be used, with placeholders, so SQL
507injection attacks will not work, but it's easier to illustrate as though the
508values were interpolated directly. Don't worry, they're not.)
509
510With the same idea in mind, you can check if a value is NULL with:
511
512 { foo => undef }
513
514This will be correctly rewritten to C<foo IS NULL>.
515
516You can pass an empty hashref if you want all rows, e.g.:
517
518 database->quick_select('mytable', {});
519
520... is the same as C<"SELECT * FROM 'mytable'">
521
522If you pass in an arrayref as the value, you can get a set clause as in the
523following example:
524
525 { foo => [ 'bar', 'baz', 'quux' ] }
526
527... it's the same as C<WHERE foo IN ('bar', 'baz', 'quux')>
528
529If you need additional flexibility, you can build fairly complex where
530clauses by passing a hashref of condition operators and values as the
531value to the column field key.
532
533Currently recognized operators are:
534
535=over
536
537=item 'like'
538
539 { foo => { 'like' => '%bar%' } }
540
541... same as C<WHERE foo LIKE '%bar%'>
542
543=item 'gt' / 'ge'
544
545 'greater than' or 'greater or equal to'
546
547 { foo => { 'ge' => '42' } }
548
549... same as C<WHERE foo E<gt>= '42'>
550
551=item 'lt' / 'le'
552
553 'less than' or 'less or equal to'
554
555 { foo => { 'lt' => '42' } }
556
557... same as C<WHERE foo E<lt> '42'>
558
559=item 'eq' / 'ne' / 'is'
560
561 'equal' or 'not equal' or 'is'
562
563 { foo => { 'ne' => 'bar' } }
564
565... same as C<WHERE foo != 'bar'>
566
567=back
568
569You can also include a key named 'not' with a true value in the hashref
570which will (attempt) to negate the other operator(s).
571
572 { foo => { 'like' => '%bar%', 'not' => 1 } }
573
574... same as C<WHERE foo NOT LIKE '%bar%'>
575
576If you use undef as the value for an operator hashref it will be
577replaced with 'NULL' in the query.
578
579If that's not flexible enough, you can pass in your own scalar WHERE clause
580string B<BUT> there's no automatic sanitation on that - if you suffer
581from a SQL injection attack - don't blame me!
582Don't forget to use C<quote()>/C<quote_identifier()> on it then.
